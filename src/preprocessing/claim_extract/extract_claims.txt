You extract reviewer-salient CLAIMS from a research paper.

Your output is not a summary of the paper. It is the set of author assertions
that an expert reviewer would need to audit because they affect novelty,
validity, contribution, reproducibility, or scope.

Return STRICT JSON only, matching this schema:

{{
  "core_claims": [
    {{
      "id": "core_claim_01",
      "text": "<concise high-level reviewer-facing claim>",
      "type": "empirical | methodological | theoretical",
      "scope": "broad | local",
      "importance": "primary",
      "datasets": ["FB15k-237", "..."],
      "baselines": ["TransE", "..."],
      "metrics": ["MRR", "..."],
      "location": {{"section_id": "<id>", "char_start": <int>, "char_end": <int>}},
      "evidence_targets": ["claim_03", "table_1.MRR", "literature.agent_skill_retrieval"]
    }}
  ],
  "claims": [
    {{
      "id": "claim_01",
      "text": "<concise verbatim-or-near-verbatim author claim>",
      "type": "empirical | methodological | theoretical",
      "scope": "broad | local",
      "importance": "primary | secondary",
      "datasets": ["FB15k-237", "..."],
      "baselines": ["TransE", "..."],
      "metrics": ["MRR", "..."],
      "location": {{"section_id": "<id>", "char_start": <int>, "char_end": <int>}},
      "evidence_targets": ["table_1.MRR", "literature.agent_skill_retrieval"]
    }}
  ]
}}

Two-layer output:

- `core_claims` is the compact display layer for the report overview and
  teaser figure. Return about 2-4 claims, normally 3, unless the paper truly
  has fewer or more central claims. These should be the paper's main
  reviewer-facing assertions: the strongest positioning/novelty assertion,
  the central method or artifact contribution, and the headline empirical or
  causal result. They may synthesize several detailed claims, but must remain
  neutral author assertions rather than your judgment.
- `claims` is the complete detailed audit layer. Return every
  reviewer-salient author claim that should be checked downstream. This layer
  has no fixed length.
- Construct `claims` first, then select or synthesize `core_claims` from that
  complete detailed set. Do not drop, merge away, or omit a detailed claim
  merely because a broader core claim already covers the same topic.
- Do not make `core_claims` more favorable than the paper warrants. If the
  paper positions itself as first/novel but related work may already cover the
  idea, still extract that positioning as a claim; do not present it as proven
  novelty.
- Every core claim should correspond to one or more detailed claims by meaning.
  Use detailed claim ids in `evidence_targets` when helpful.

What counts as a reviewer-salient claim:

- Novelty, first-work, uniqueness, or literature-gap positioning.
- A central method or design contribution, only when the paper presents it as
  a contribution or when it is necessary to evaluate the claimed result.
- Artifact/resource claims that matter to the contribution, such as corpus
  scale, provenance, availability, or benchmark construction.
- Main empirical effect claims, including improvements, null results,
  efficiency/cost claims, ceilings, and statistically tested comparisons.
- Causal or explanatory claims drawn from results, such as what the paper says
  is the bottleneck, mechanism, or reason for success/failure.
- Generalization, coverage, reproducibility, or practical-impact claims.

What should not be extracted:

- Neutral descriptions of the system architecture or pipeline mechanics unless
  the paper asserts novelty, evidence, or contribution through that description.
- Routine settings, hyperparameters, model names, truncation lengths, top-k
  values, prompt templates, implementation details, or benchmark descriptions
  unless the paper makes them part of a reviewed contribution or result.
- Background/motivation facts and generic related-work summaries, except for
  explicit gap or positioning claims made by the authors.
- Figure/table captions as separate claims when they only restate a claim that
  already appears in the abstract, contribution block, or main results text.
- Repeated restatements of the same assertion from different sections.
- Supporting measurements that only justify a broader takeaway in the same
  paragraph or section. Put those measurements in metrics/evidence_targets of
  the broader claim instead of returning separate claims.

Granularity:

- There is no fixed number of claims. Include every reviewer-salient author
  claim in `claims` and omit low-salience descriptive facts.
- One returned claim should have one main verifiable predicate.
- Split claims that require different evidence. For example, split resource
  size from public availability; score improvement from oracle-ceiling ratio;
  skill-use rate from performance gain; causal interpretation from the result
  it interprets; retrieval coverage from agent use rate; method existence
  from first/novel positioning.
- When extracting novelty or positioning, do not keep extra artifact, corpus,
  benchmark, or implementation clauses inside that novelty claim. Extract the
  novelty assertion itself, then extract the other assertions separately if
  they are reviewer-salient.
- For artifact/resource claims, keep availability, provenance/construction,
  and exact size as separate claims when they are all reviewer-salient. Do not
  repeat the exact size inside the availability claim if a size claim is also
  returned.
- Deduplicate by meaning, not wording. If the same assertion appears in the
  abstract, introduction, contribution block, and discussion, keep the clearest
  contribution-block or abstract wording once.
- If one claim already captures the reviewed takeaway, do not also return each
  table row, stage metric, sub-metric, or implementation-detail sentence that
  merely supports it.
- If two returned claims would cite essentially the same evidence and have the
  same reviewer consequence, merge them into the clearer higher-level claim.
  If they require different evidence, split them.
- Preserve the authors' assertion neutrally. Do not make it stronger, weaker,
  more favorable, or more skeptical; downstream audit will judge support.
- `core_claims` can be broader than detailed claims because it is a display
  layer, but it should not become a vague summary. Each core claim still needs
  a concrete reviewed predicate and a trace to detailed claims or manuscript
  evidence.

Granularity examples:

- Too broad: "We present the first X method over a 36K corpus and improve
  Pass@1." Better: "This is the first X method"; "The corpus contains 36K
  items"; "The method improves Pass@1 from A to B."
- Too broad: "The method improves Pass@1 and reaches 84% of the oracle
  ceiling." Better: one claim for the improvement, one claim for the oracle
  ceiling.
- Too broad: "Agents use retrieved skills but performance does not improve."
  Better: one claim for use rate, one claim for null performance gain.
- Too broad: "The system retrieves 61.8% of oracle items and agents use them
  at 68.8%." Better: one claim for retrieval coverage, one claim for agent use.
- Too fine: "Stage 1 has R@1000 of 0.905; Stage 2 has MRR 0.587; Stage 3
  has MRR 0.634." Better: "Each stage progressively improves ranking quality"
  unless one stage's number is itself a headline contribution.

Before returning JSON, silently check each detailed claim: if a reviewer would
need two different evidence checks to verify it, split it. If a detailed claim
is only evidence for another returned detailed claim, remove it and attach its
evidence target there. Then check that `core_claims` is a compact neutral
display layer over the detailed claims, not a second exhaustive list.

Type guidance:

- empirical: numeric result, comparison, statistical test, null result,
  efficiency/cost/latency claim, benchmark or artifact statistic.
- methodological: proposed method/framework/design/resource construction.
- theoretical: conceptual framing, explanatory/causal interpretation,
  generality, or literature-positioning claim that is not itself numeric.

Use "primary" for core novelty/contribution/result/causal claims. Use
"secondary" for supporting claims that a reviewer may still need to check.

Return {{"core_claims": [], "claims": []}} only if there are no
reviewer-salient author claims.
